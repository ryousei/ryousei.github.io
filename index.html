Improving Server Application Performance via Pure TCP ACK Receive Optimization

  • Posted on: Dec 10 2014
  • By: Ryousei Takano
  • With:

Title: Improving Server Application Performance via Pure TCP ACK Receive Optimization
Authors: Michael Chan and David R. Cheriton, Stanford University
2013 USENIX ATC [URL]

  • The authors present TCP-PARO, (Pure ACK Receive Optimization), that removes the bulk of expensive per-packet processing and provides a fast processing path of pure ACKs in the network stack (bypasses the original stack).
    • Pure ACK parsing and demultiplexing
    • ACK processing fast module
    • Handling a mixture of pure ACKs and other TCP segments
  • The authors show that ACK receive processing can consume up to 20% CPU cycles in a Web server. Using this technique, they demonstrate cycles savings of 15%, and 33% throughput improvement in reliable multicast, TCP-SMO [INFOCOM2002].
  • Some batching optimization have been proposed
  • By the way, the last author is famous as his work, V operating system, in 80s.

Lightweight Memory Tracing

  • Posted on: Dec 9 2014
  • By: Ryousei Takano
  • With:

Title: Lightweight Memory Tracing
Authors: Mathias Payer, Enrico Kravina, and Thomas R. Gross, ETH Zurich
2013 USENIX ATC [URL]

  • The authors present memTrace, a lightweight memory tracing technique that builds on dynamic on-the-fly cross-ISA binary translation of 32-bit code to 64-bit code.
  • Current software-only memory tracing incurs high performance overhead (up to 10x for libdft [VEE2012], 20x for Valgrind’s memcheck [PLDI2007], and up to 40x for taintcheck [NDSS2005]). This is because every single memory access of the application is checked by additional code (memlet) and hardware is limited to a small set of watched locations (watchpoints).
  • MemTrace dynamically translates 32-bit x86 application code to x64 code (i.e., 64-bit code), and that enables memlets, code sequences that are woven into the target application code to execute memory tracing, to use extra registers and a huge shadow memory space (64-bit address space).
  • The authors demonstrates two memory tracing case studies: unlimited watchpoints and safe heap memory allocator.
  • The prototype implementation extends the libdetox binary translation platform [VEE2011].
  • The source code is available from gannimo/memTrace@github.

ClickOS and the Art of Network Function Virtualization

  • Posted on: Dec 8 2014
  • By: Ryousei Takano
  • With:

Title: ClickOS and the Art of Network Function Virtualization
Authors: Joao Martins and Mohamed Ahmed, NEC Europe Ltd.; Costin Raiciu and Vladimir Olteanu, University Politehnica of Bucharest; Michio Honda, Roberto Bifulco, and Felipe Huici, NEC Europe Ltd.
2014 USENIX NSDI [URL]
Reference:

  • Cloud Networking Platform Lab (NEC Europe)
  • The authors presents ClickOS, a high-performance, virtualize software middlebox platform.
  • To support fast and easily programmable middlebox platform, ClickOS implements a minimalistic guest OS that is organized to run Click, a software modular router, at rates of millions of packets per second. It is built on top of MiniOS in the Xen hypervisor. The authors also introduce the VALE switch [CoNEXT2012] instead of Open vSwitch as the backend switch. To do that, they modified the Xen netback driver to implement the netmap API.
  • To demonstrate the feasibility, the authors test the following middlebox applications: Wire, EtherMonitor, IP Router, Firewall, Carrier Grade NAT, Software Broadband Remote Access Server, Intrusion Detection System, Load Balancer, and Flow Monitor.
  • The source code is available from cnplab@github.

Title: Speeding up packet I/O in virtual machines
Authors: Luigi Rizzo, Giuseppe Lettieri, Vincenzo Maffione, Università di Pisa, Italy
2013 IEEE/ACM ANCS [URL]
Reference:

  • The netmap project (Università di Pisa)
  • The authors applies netmap and other techniques to QEMU achieving 5 Mpps with netmap [USENIX2012], 1 Mpps with sockets between VMs over an emulated e1000.
    • interrupt moderation
    • send combination [USENIX2002]: a driver-based form of batching and interrupt moderation
  • They focus on high packet rates instead of high throughput on virtualized networks. That enables a new set of applications, including software routers, middlebox, SDN, and so on, within virtual machines.
  • They also use VALE [CoNEXT2012] as a virtual switch.

netmap: a novel framework for fast packet I/O

  • Posted on: Dec 6 2014
  • By: Ryousei Takano
  • With:

Title: netmap: A Novel Framework for Fast Packet I/O
Authors: Luigi Rizzo, Università di Pisa, Italy
2012 USENIX ATC [URL] (Best paper award)
Reference:

  • The netmap project (Università di Pisa)
  • The author presents netmap, a novel framework that enables commodity operating systems to handle the millions of packets per second (pps), without requiring custom hardwares and changes to applications.
  • In order to build network applications like network monitors and packet generators, UNIX provides raw packet I/O APIs, e.g., Berkeley Packet Filter (BPF) and AF_SOCKET, however, the performance is inadequate for the millions of pps on a 10 gigabit link.
  • netmap integrates some well-known existing ideas into commodity OSs to reduce the cost of moving traffic between the NIC and the network stack. For example, NetChannels is a similar idea, and it is employed in OSv [USENIX2014].
    • per-packet dynamic memory allocations, removed by preallocating resources
    • system call overheads, amortized over large batches
    • memory copies, eliminated by sharing buffers and metadata between kernel and userspace
  • The following snippet is an example code of packet generator using netmap API.
  • netmap is orthogonal to hardware offloading (checksum, TSO, LRO) and virtualization (VMDc, SR-IOV) mechanisms. The authors have been proposed a netmap-based software switch, VALE [CoNEXT2012] [ANCS2013].
  • netmap has been implemented in FreeBSD and Linux. The recent FreeBSD has already included it, and also the latest implementation is available from Google code page.

mTCP: a Highly Scalable User-level TCP Stack for Multicore Systems

  • Posted on: Dec 5 2014
  • By: Ryousei Takano
  • With:

Title: mTCP: a Highly Scalable User-level TCP Stack for Multicore Systems
Authors: EunYoung Jeong, Shinae Woo, Muhammad Jamshed, and Haewon Jeong, Korea Advanced Institute of Science and Technology (KAIST); Sunghwan Ihm, Princeton University; Dongsu Han and KyoungSoo Park, Korea Advanced Institute of Science and Technology (KAIST)
2014 USENIX NSDI [URL] (Community award)
Reference:

  • mTCP (KAIST)
  • The authors presents mTCP, a highly scalable user-level TCP stack for multicore systems.
  • Large flows are easy to fill up 10 Gbps. But, small flows are hard to fill up 10 Gbps regardless of the number of CPU cores. This is because the Linux kernel is not designed for multicore systems. In a web server case, 83% of CPU usage spent inside kernel.
  • Several proposals for improving in-kernel TCP performance, including affinity accept [EuroSys2012], MegaPipe [OSDI2012], and FlexSC [OSDI2010], however, they are not sufficient. As another approach, mTCP, SolarFlare OpenOnload and Sandstorm [SIGCOMM2014] run the entire network stack in the userspace to eliminate kernel crossing overheads and optimize packet processing without kernel modifications.
  • mTCP makes full use of extreme parallelism and batch processing:
    • Per-core resource management (listening queue, file descriptor, packet queue)
    • Lock-free data structures and cache-aware threading
    • Eliminate system call overhead
    • Reduce context switch cost by event batching
    • Easily portable APIs for compatibility (similar to BSD socket API)
  • This group has proposed PacketShader, GPU-accelerated software router in the ACM SIGCOMM 2010. As a part of mTCP (user-level pacekt I/O library), they extend PacketShader I/O engine (PSIO) to support an efficient event-driven packet I/O interface. Other user-level packet I/O engines have been proposed, including Intel DPDK and netmap [USENIX2012].
  • The source code is available from eunyoung14@github.

OSv - Optimizing the Operating System for Virtual Machines

  • Posted on: Dec 4 2014
  • By: Ryousei Takano
  • With:

Title: OSv - Optimizing the Operating System for Virtual Machines
Authors: Avi Kivity, Dor Laor, Glauber Costa, Pekka Enberg, Nadav Har’El, Don Marti, and Vlad Zolotarov, Cloudius Systems
2014 USENIX ATC [URL]
Reference:

  • OSv - the operating system designed for the cloud
  • The authors present OSv, a new guest operating system designed specifically for running a single application on a virtual machine in the cloud. The first author is famous as the father of the KVM hypervisor and a founder at Cloudius Systems.
  • OSv follows the library OS design, proposed by exokernel in 1990s. In OSv case, a hyperviosr replaces the role of exokernel.
  • OSv has been developed from scratch. The code is written in C++11.
    • Single address space: all application thread and kernel use the same page tables.
    • No spinlocks.
    • Thread scheduler: lock-free, preemptive, tick-less, fair, scalable, and efficient
    • Network channels: employ Van Jacobson’s net channel idea to reduce contentions between an application thread context and an interrupt context.
    • Beyond Linux API: shrinker, JVM balloon
  • It originally supports only Java applications, and now applications written in Lua, Ruby, and so on, run on OSv. Some guest OSs focus on running only a single high-level language. For example, Erlang on Xen, Mirage OS (OCaml), and Libra (J9 JVM).

Hyper-Switch: A Scalable Software Virtual Switching Architecture

  • Posted on: Dec 3 2014
  • By: Ryousei Takano
  • With:

Title: Hyper-Switch: A Scalable Software Virtual Switching Architecture
Authors: Kaushik Kumar Ram, Alan L. Cox, Mehul Chadha, and Scott Rixner, Rice University
2013 USENIX ATC [URL]

  • This paper presents Hyper-Switch, a efficient and scalable software-based switch, and several optimization techniques, including preemptive packet copying, batching hypervisor entries, and offloading packet processing.
  • Benson et al. reports that 80% of the traffic originating at servers in cloud data centers never leaves the rack [IMC2010]. Also the number of CPU cores in a server will grow continuously. These implies that the role of vSwich becomes much more critical. Hyper-Switch focuses on the performance problem of the last hop in a virtualized data center.
  • The prototype is implemented based on Open vSwitch under the Xen 4.2 platform. The authors have ported essential parts of Open vSwitch’s datapath (from the driver domain) to the Xen hypervisor.
  • The experimental results show the advantage over unmodified Xen and KVM with vhost-net.

vTurbo: Accelerating Virtual Machine I/O Processing Using Designated Turbo-Sliced Core

  • Posted on: Dec 2 2014
  • By: Ryousei Takano
  • With:

Title: vTurbo: Accelerating Virtual Machine I/O Processing Using Designated Turbo-Sliced Core
Authors: Cong Xu, Sahan Gamage, Hui Lu, Ramana Kompella, and Dongyan Xu, Purdue University
2013 USENIX ATC [URL]
Reference:

  • Improving VM Network I/O Performance (Purdue Univ.)
    • vSnoop, vFlood, vSlicer, vPipe, …
  • The authors present vTurbo, a system that offloads I/O processing for VMs to a designated core with small scheduling time-slice. It enables us to reduce both VM scheduling latency and low I/O IRQ latency.
  • This system divides CPU cores into regular cores (normal CPU time-slice, e.g., 30ms) and turbo cores (small CPU time-slice, e.g., 0.1ms). I/O processing for VMs is assigned to one or more turbo core(s). A turbo vCPU of each VM is pinned to the corresponding turbo core.
  • The prototype is implemented based on Xen 4.1.2.
  • For improving TCP performance, it uses a technique similar to their previous work, vSnoop [SC2010], that offloads ACK processing from a guest OS to the hypervisor. Such a hypervisor intervention is similar to this work.
  • The problem statement is similar to ELVIS [USENIX2013], but ELVIS uses polling instead of thread scheduling.

Efficient and Scalable Paravirtual I/O System

  • Posted on: Dec 1 2014
  • By: Ryousei Takano
  • With:

Title: Efficient and Scalable Paravirtual I/O System
Authors: Nadav Har’El, Abel Gordon, and Alex Landau, IBM Research–Haifa; Muli Ben-Yehuda, Technion IIT and Hypervisor Consulting; Avishay Traeger and Razya Ladelsky, IBM Research–Haifa
2013 USENIX ATC [URL]
Reference:

  • Virtual I O acceleration technologies for KVM (Abel Gorden@github)
  • The authors proposes a novel para-virtual I/O mechanism, ELVIS (Efficient and scaLable paraVirtual I/O System).
    • Fine-grained I/O scheduling (instead of thread-based I/O scheduling): a single I/O thread runs on a dedicated I/O core, and handles I/O requests of multiple VMs.
    • Exitless I/O request notification: polling in host’s I/O core
    • Exitless I/O reply notification: using ELI
  • They implemented ELVIS in KVM/vhost, and demonstrated the performance of I/O intensive applications comparing to the unmodified KVM with vhost and SR-IOV+ELI.
  • ELI (Exit-Less Interruption) is their previous work [ASPLOS2012], and improves interrupt virtualization efficiency. In this paper, they emulate posted interrupts, which is a new feature announced for future x86 processors, by using ELI.
    • VT-d Posted Interrupts
    • Add VT-d Posted-Interrupts support for KVM (LWN)
